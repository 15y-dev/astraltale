🔑 ハッシュアルゴリズム

FNV-1 64bit 変種 ✅ Ghidra 解析で確定

# FUN_140d023a0: 1バイトハッシュ
def hash_char(byte, seed):
    return (seed * 0x1000193 ^ byte) & 0xFFFFFFFFFFFFFFFF

# FUN_140d023c0: ファイルハッシュ計算
def calc_hash(filename, path):
    h = 0
    for b in filename.lower():  h = hash_char(b, h)
    for b in path.lower():      h = hash_char(b, h)
    return h

パス正規化ルール (FUN_140d02b60)

  • \ → / に変換
  • // → / に正規化
  • 先頭の ./ を除去
  • 末尾の / を除去
  • 全て小文字に変換
  • ハッシュは filename + path の順で計算（path が後）

📂 FileList とNFSファイル解決

なぜ複数の FileList が必要か？

ゲームのファイル解決は2段階で行われる:

① packageindex  →  「このハッシュのデータは offset=XX, size=YY にある」
                     ただし どの NFS ファイルに入っているかは不明

② FileList      →  「このハッシュ → 'dae933ca' という NFS ファイルにある」
                     NFS ファイルパス: nfs/{先頭1文字}/{名前}

packageindex だけではデータの場所（offset/size/checksum）は分かるが、物理的にどの NFS ファイルを開けばいいかが分からない。FileList がハッシュ → NFS ファイル名のマッピングを提供する。

FileList の種類

ファイル内容エントリ数
FileListPC.txt メインゲームデータ（c_*.ini, モデル, テクスチャ等） ~79,457
GameDataTranslateFileList_jp.txt 日本語翻訳データ（t_*_jp.ini） ~35
GameDataTranslateFileList_en.txt 英語翻訳データ（t_*_en.ini） ~35
GameDataTranslateFileList_cn.txt 中国語翻訳データ（t_*_cn.ini） ~35

FileList エントリ形式

# ハッシュ行（翻訳ファイル等）:
0ea3e3b9c71e6995,dae933ca,6a2a8edf,10842,275443,8b8bfb62,83abc5c4,00000000
│                   │        │         │     │      │        │        └ 固定値
│                   │        │         │     │      │        └ checksum
│                   │        │         │     │      └ CRC32
│                   │        │         │     └ 展開後サイズ
│                   │        │         └ 圧縮サイズ
│                   │        └ タイムスタンプ
│                   └ NFS ファイル名 → nfs/d/dae933ca
└ ファイルハッシュ (64bit hex)

NFS ファイル名の解決フロー

hash = calc_hash("t_node_jp.ini", "data/db")  → 0x0EA3E3B9C71E6995

# 1. FileList で NFS名を検索
filelist[0x0EA3E3B9C71E6995] → nfs_name = "dae933ca"

# 2. NFS ファイルを開いてチャンクを読む
path = nfs/d/dae933ca    ← 先頭1文字がサブディレクトリ
offset = packageindex[hash].offset
data = extract_chunk(path, offset, size, checksum)
