暗号の特性分析

  • パディングなし — 1バイト増えるとパケットも1バイト増える → ブロック暗号ではない
  • 同一平文で異なる暗号文 — 「a」を2回送ったが完全に別の暗号文 → 単純XORではない
  • 最有力候補: RC4 — ストリーム暗号、ゲームで最も一般的、状態が進むため毎回異なる出力
  • 他の候補 — ChaCha20、AES-CTR（いずれもストリーム暗号的に動作）

🔒 最終結論

パケット復号: 不可能

パッシブなパケットキャプチャのみでは、通信内容の復号は不可能です。

理由

  • RC4 セッション鍵（5バイト）は 公開鍵暗号（RSA/ECC） でラップされてサーバーに送信される
  • サーバーの秘密鍵がなければ、パケットから RC4 鍵を復元できない
  • セッションごとに鍵が変わる（rdtsc ベースの乱数）ため再利用も不可

復号に必要な手段（いずれも利用不可）

手段内容可否
メモリダンププロセスメモリから [+0x78] の5バイト鍵を読み取り❌ 不可
DLL インジェクションKSA 関数にフックして鍵をダンプ❌ 不可
デバッガ (x64dbg)KSA にブレークポイントで鍵を読む❌ 不可
バイナリパッチ公開鍵を差し替えて自前で復号❌ 不可

📊 鍵交換プロトコル

1. クライアント: WSAConnect → サーバーに TCP 接続
2. クライアント: rdtsc → 5バイト RC4 鍵生成 → KSA で送信/受信ステート初期化
3. サーバー → クライアント: 8バイトヘッダ + 公開鍵パート1 + 公開鍵パート2
4. クライアント: LibTomCrypt で公開鍵を暗号コンテキストに設定
5. クライアント → サーバー: 公開鍵暗号で暗号化された RC4 鍵 📡
6. サーバー: 秘密鍵で復号 → 同じ RC4 鍵で送受信ステート初期化
7. 以降: RC4 で通信 (送信 [+0x80], 受信 [+0x88])
